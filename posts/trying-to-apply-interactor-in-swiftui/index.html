Trying to Apply Interactor in SwiftUI

When we talk about SwiftUI, we must first understand that the patterns applied to UIKit can‚Äôt be directly applied in this new approach. The problems we need to solve when we use UIKit are different from the ones when we use SwiftUI, and trying to apply the same solutions as it is to both seems to be a lazy approach that will lead you to headaches in the future.

But I will not be a hypocrite here. It‚Äôs inevitable that, as human beings, we make connections between what we already know to anything we are trying to learn. It‚Äôs a natural and really useful tool helping us to learn faster and to make brain connections that will last longer. That‚Äôs the reason you‚Äôll find many online articles entitled ‚ÄúApplying MVC in SwiftUI‚Äù, ‚ÄúSwiftUI with MVVM‚Äù, or any other architecture broadly applied in the UIKit development world, like VIPER, VIP, and so on. These articles are the result of people trying to make connections between what they know to what they are learning.

During my recent experience with SwiftUI, for an indie app I‚Äôve been developing, I was trying to find an approach to handle persistence encapsulation. My first solution was to persist in UserDefaults, but using an architecture approach where when I need to replace UserDefaults for a better solution, I can replace the integration just in one place in the code.

Interactors

During my research for a solution, I found out Interactors, a component broadly used in VIPER and VIP architectures. It‚Äôs an interface between model or data persistence layers and the UI modules that, in my case, were the views and view models. I decided to use ViewModel for my project as a suggestion from a SwiftUI Apple engineer during a lab at WWDC2020, where he assisted me with my project architecture.

Something we need to understand when we‚Äôre developing our project is that all patterns are adaptable. We don‚Äôt need to implement VIP to use Interactor. We can adapt this component to what we need, and in most cases, we will find components that solve the same problem being used in different architectures with different names. For example, an Interactors is just an Interface, like in Java, where we provide a set of behavior and whoever will conform to that Interface must implement these behaviors. And guess what? A Protocol in Swift can be seen as an Interface in Java, and so on. Of course, each approach has its nuances, but the purpose of usage is similar.

Enough Talk! Show me the code

I will share my implementation for Interactors in my project, but you should understand that it should not be used directly in yours. You are welcome to adapt it if you need, but you must understand that your project is unique, reflecting on what you are coding.

MainInteractor

This is a protocol to be conformed from anyone who wants to become an Interactor:

protocol MainInteractor {
    var appState: AppState { get }
    func loadAppState()
    func saveAppState()
}

I decided to save and load the app state as a whole in one operation. But remember: this is just the interface. If I need, I can make a complex rule to persist and load only the changed data without changing this interface. That’s the beauty of encapsulation.

My AppState is just a model that keeps the data I need to persist. As it is not relevant for this article, I will show an example below:

import SwiftUI

public class AppState: ObservableObject {
    @Published var user = User()
}

MainInteractorImp

MainInteractorImp is my component that conforms to the protocol MainInteractor:

struct MainInteractorImp: MainInteractor {
    
    let userRepository: AppStateRepository
    let appState: AppState
    
    init(userRepository: AppStateRepository, appState: AppState) {
        self.userRepository = userRepository
        self.appState = appState
    }
    
    func loadAppState() {
        appState.user = userRepository.loadUser()
    }
    
    func saveAppState() {
        userRepository.saveUser(appState.user)
    }
}

AppStateRepository is just a wrapper that is responsible for the persistence logic. As the Interactor should not only be used for persistence, and in the future, I may give it other responsibilities, like networking calls, I want to be able to replace only the AppStateRepository for only persistence changes.

InteractorsContainer

InteractorsContainer is a singleton that wraps the Interactor used in the project. It‚Äôs the element that will be integrated with the SwiftUI, allowing us to change almost every other layer explained above without needing to change any code inside the SwiftUI views.

import SwiftUI

struct InteractorsContainer: EnvironmentKey {
    
    let mainInteractor: MainInteractor
    
    static var defaultValue: Self { Self.default }
    
    private static let `default` = Self(mainInteractor: MainInteractorImp(userRepository: UserDefaultsAppStateRepository(), appState: AppState()))
    
    init(mainInteractor: MainInteractor) {
        self.mainInteractor = mainInteractor
        self.mainInteractor.loadAppState()
    }
}

extension EnvironmentValues {
    var interactors: InteractorsContainer {
        get { self[InteractorsContainer.self] }
        set { self[InteractorsContainer.self] = newValue }
    }
}

InteractorsContainer is initialized in SceneDelegate to load data when the app starts, and it sets the app state as an environment object, like below:

func scene(_ scene: UIScene, willConnectTo session: UISceneSession, options connectionOptions: UIScene.ConnectionOptions) {
        let interactors = InteractorsContainer.defaultValue
        interactors.mainInteractor.loadAppState()

		let myView = CustomView().environmentObject(interactors.mainInteractor.appState)
		// ...
}

Inside SwiftUI views, we will define two parameters:

struct OnboardingAvatarPickerView: View {
    
    @EnvironmentObject var appState: AppState
    @Environment(\.interactors) var interactors: InteractorsContainer
    
    var body: some View {
		//...
	  }
}

Now, whenever we want to persist some data, we just call self.interactors.mainInteractor.saveAppState() in the View. For example, I have an EditView, where the user changes the app configuration. Then I call this persistence method inside .onDisappear, like below:

var Body: some View {
	MyView {
		// any view here
	}.onDisappear {
  		self.interactors.mainInteractor.saveAppState()
  }
}
